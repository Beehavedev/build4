─────────────────────────────────────────
1. WALLET SERVICE (src/services/wallet.ts)
─────────────────────────────────────────

- generateEVMWallet(): uses ethers.Wallet.createRandom(), returns {address, privateKey}
- generateSolanaWallet(): uses Keypair.generate(), returns {address, privateKey as base58}
- encryptPrivateKey(pk, userId): AES-256-CBC with key = MASTER_KEY + userId hash
- decryptPrivateKey(encrypted, userId): reverse of above
- getBalance(address, chain): returns USDT + native token balance
- NEVER store raw private keys. Always encrypt immediately on generation.
- The /wallet command shows: address (truncated + copy button inline keyboard), chain balances, option to generate new or switch active.
- Private key export: require user to type "I CONFIRM" in chat, log to audit table, show once then delete message after 30 seconds.

─────────────────────────────────────────
2. TRADING AGENT (src/agents/tradingAgent.ts)
─────────────────────────────────────────

The agent tick function runs every 60 seconds via cron for all active agents.

Each tick does this in order:

STEP 1 — GATHER CONTEXT
Fetch: current price + 1h/4h/1d OHLCV for agent's pairs, open positions on exchange, current margin/balance, last 10 agent memories, user's risk settings, today's PnL vs maxDailyLoss.

STEP 2 — RISK GUARD CHECK (riskGuard.ts)
Before calling AI, check:
- Has daily loss limit been hit? → pause agent, notify user
- Is there already an open position in this pair? → skip unless adding/reducing
- Is position size within limits? → clamp if needed
- Is leverage within user's max? → reject if exceeded
If any hard block, do NOT call Claude. Log the block reason.

STEP 3 — AI DECISION (Claude call)
Call Claude claude-sonnet-4-5 with this system prompt:

"""
You are an expert crypto perpetual futures trading agent. You analyze market data and make precise trading decisions. You must respond ONLY with valid JSON matching the schema exactly.

Your personality: disciplined, risk-aware, patient. You prefer high-probability setups over frequent trades. You never revenge trade. You always respect stop losses.

You have access to this agent's memory which contains observations about past trades, market patterns you've noticed, and corrections from previous mistakes. Use this memory to improve decisions.

Current agent config:
- Max position size: {maxPositionSize} USDT
- Max leverage: {maxLeverage}x
- Stop loss: {stopLossPct}%
- Take profit: {takeProfitPct}%
- Max daily loss: {maxDailyLoss} USDT
- Today's PnL so far: {todayPnl} USDT

Respond with exactly this JSON schema:
{
  "action": "OPEN_LONG" | "OPEN_SHORT" | "CLOSE" | "HOLD",
  "pair": "BTC/USDT",
  "size": 100,
  "leverage": 3,
  "stopLoss": 42000,
  "takeProfit": 44000,
  "confidence": 0.72,
  "reasoning": "Plain English 2-3 sentence explanation of WHY you're taking this action. Mention specific signals. Be honest about risks.",
  "memoryUpdate": "One sentence observation to remember for future ticks, or null"
}

If action is HOLD, still provide reasoning explaining what you're waiting for.
"""

User message contains: formatted price data, OHLCV summary, current positions, recent memories.

STEP 4 — EXECUTE
If action is OPEN_LONG/OPEN_SHORT:
- Call exchange API (Aster or Hyperliquid)
- Save Trade record with aiReasoning from Claude
- Save memoryUpdate to AgentMemory table
- Push Telegram notification to user:
  "🤖 Agent {name} opened {side} {pair}
   Entry: ${price} | Size: ${size} | {leverage}x
   SL: ${sl} | TP: ${tp}
   
   💭 Why: {reasoning}
   
   Confidence: {confidence*100}%"

If action is CLOSE: close position, calculate PnL, update Trade record, notify user with PnL result.

If action is HOLD: log reasoning to memory (skip if confidence < 0.3), no notification.

STEP 5 — PORTFOLIO UPDATE
Recalculate portfolio totalValue, dayPnl, totalPnl after every executed action.

SECURITY REQUIREMENTS — NON-NEGOTIABLE
═══════════════════════════════════════════

1. Private keys MUST be AES-256 encrypted before any DB write. The decrypted key must never appear in logs.
2. All Telegram user IDs must be validated against DB before any action.
3. Rate limiting: max 30 commands per minute per user via Redis.
4. Agent runner must check circuit breaker (maxDailyLoss) before EVERY trade execution — no exceptions.
5. Contract scanner results must never be cached beyond 5 minutes (contract state changes).
6. All Claude API calls must have a 30-second timeout and graceful fallback if they fail.
7. Wrap every exchange API call in try/catch — a failed trade must NEVER crash the agent runner.

IMPORTANT NOTES FOR REPLIT AGENT
═══════════════════════════════════════════

- Use grammY (grammy npm package), NOT telegraf. They have different APIs.
- For Grammy conversations use @grammyjs/conversations plugin.
- For Grammy sessions use @grammyjs/storage-redis with Upstash.
- Prisma with PostgreSQL on Replit: use the built-in PostgreSQL add-on and set DATABASE_URL from the Replit secrets panel.
