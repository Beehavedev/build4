New Findings — Post-Patch Regression (6)

These vulnerabilities were introduced after the security patches in 20 new feature commits (Aster V3 Direct connection, Telegram MiniApp updates). They represent a security regression.

NEW-01 CRITICAL REGRESSION Cloudflare WAF Bypass for All MiniApp Endpoints
server/index.ts:66

The Cloudflare enforcement middleware now exempts all paths matching /miniapp, /miniapp-old, and /api/miniapp/*. This includes security-critical financial endpoints: /api/miniapp/trade (order execution), /api/miniapp/link-aster (private key submission), /api/miniapp/close-position, and all balance/position queries.

Impact: Attackers can directly reach financial endpoints without Cloudflare DDoS/WAF protection, enabling brute-force attacks and direct exploitation.

Recommendation: Only bypass Cloudflare for the HTML page (/miniapp), not the API routes. Apply rate limiting specifically to /api/miniapp/*.

NEW-03 CRITICAL REGRESSION User Wallet Private Keys Stored as V3 Direct Signer
server/miniapp-routes.ts:156-170, server/telegram-bot.ts:13380-13395

The new "V3 Direct" connection flow uses the user's main wallet private key as the Aster signer. This private key is stored encrypted in the aster_credentials table as encrypted_api_secret. Previously, only API keys (revocable) were stored.

Impact: If the database is compromised (see C12 — deployer key in .replit), ALL user wallet private keys can be extracted and funds drained. Unlike API keys, private keys cannot be revoked.

Recommendation: Never store main wallet private keys. Use API Wallet delegation (separate key pair) exclusively. If V3 Direct requires wallet signatures, implement a session-based signing flow.

NEW-02 MEDIUM X-Frame-Options Removed for MiniApp Routes
server/index.ts:201-204

X-Frame-Options: DENY and Cross-Origin-Opener-Policy: same-origin are conditionally removed for miniapp routes to allow Telegram iframe embedding. While the CSP frame-ancestors directive restricts framing to web.telegram.org and *.telegram.org, older browsers may not support CSP frame-ancestors.

Recommendation: Set X-Frame-Options: ALLOW-FROM https://web.telegram.org as a fallback for legacy browsers.

NEW-04 MEDIUM API Response Body Logged in Production
server/aster-client.ts:241

The Aster HMAC request handler now logs the first 500 characters of every API response: body=${text.substring(0, 500)}. This can include user balances, positions, account details, and potentially sensitive order information in production logs.

Recommendation: Remove body logging or restrict to non-production environments. Use structured logging with PII redaction.

NEW-05 MEDIUM parentAddress Spoofable via Client Request Body
server/miniapp-routes.ts:132-136

The /api/miniapp/link-aster endpoint accepts a user-supplied parentAddress from the request body with only basic format validation (starts with 0x, length 42). No signature verification proves the caller owns that address. An attacker could link their API wallet to another user's parent address.

Recommendation: Require an EIP-712 signature from the parent address proving wallet ownership, or derive the parent address server-side from authenticated session data.

NEW-06 LOW Silent API v1/v3 Endpoint Fallback
server/aster-client.ts — all futures methods

Every Aster futures method now wraps calls in try { v1 } catch { v3 } fallback patterns. If the v1 endpoint returns an authorization or signature error, the fallback silently retries on v3 which may have different validation rules. Some catch blocks are bare (catch {}) with no error logging.

Recommendation: Only fallback on specific error codes (e.g., 404 Not Found), not on all errors. Log all fallback events. Never silently swallow authorization errors.

Remaining Original Findings (7 Open + 2 Partial)

These findings from the initial audit remain unresolved. Smart contract issues require on-chain redeployment.

C09 CRITICAL ERC20 Transfer Return Value Ignored (BUILD4Staking)
contracts/web4/BUILD4Staking.sol:104, 136, 150, 157

ERC20 transfer() and transferFrom() return values are not checked. Tokens that return false instead of reverting will silently fail. Requires contract redeployment with SafeERC20.

C12 AGGRAVATED Deployer Private Key Hardcoded in .replit — Now Affects User Funds
.replit:56 — fbb5ce6442...ecafa87de

Originally a deployer key exposure issue. Now compounded by NEW-03: user wallet private keys are stored in the same database, encrypted with a key derived from environment variables. If the environment is compromised (key in .replit is public on GitHub), the encryption key for ALL user private keys could be derived.

Combined impact: C12 + NEW-03 = potential theft of all user funds.

Recommendation: Immediately stop storing user wallet private keys. Transfer contract ownership to a fresh multisig. Purge key from git history. Rotate ALL encryption keys.

H07HIGH creditAgent Creates Unbacked Virtual Balances
contracts/web4/AgentEconomyHub.sol:113

creditAgent() increases virtual balances without real BNB backing. Insolvency risk. Requires contract fix.

H08HIGH addRewards() Overwrites Rate, Orphans Tokens
contracts/web4/BUILD4Staking.sol:155-161

Calling addRewards() before the current reward period ends overwrites rewardRate, permanently locking undistributed tokens.

H09HIGH withdrawPlatformFees() Always Reverts
contracts/web4/SkillMarketplace.sol:123-130

Contract tracks accumulatedPlatformFees but never receives real BNB. The call{value: amount} will always revert. Dead code.

H15HIGH Complete Centralization — Single EOA Owns All Contracts
All 5 contracts — Ownable(msg.sender)

No timelock, multisig, or governance. Single compromised key = total control over all contracts.

H17PARTIAL Remaining sql.raw() Instances in storage.ts
server/storage.ts — estimated 10-15 remaining

~15 of the original 30+ sql.raw() calls were fixed with parameterized queries. An estimated 10-15 remain. New code in this version adds safe Drizzle ORM usage but does not address the legacy instances.

Conclusion & Risk Assessment

Positives

  • Rapid patch response: 5 patches within 2 hours
  • 33/42 critical/high findings from initial audit resolved
  • C04 (VM RCE) fully fixed with isolated-vm
  • 65% of token supply in on-chain vesting contracts
  • Encryption, auth, and logging significantly improved

Critical Concerns

  • NEW-03 + C12: User wallet PKs in DB + exposed deployer key = fund theft vector
  • NEW-01: Financial endpoints bypass Cloudflare WAF
  • 0% liquidity locked: LP rug pull remains possible
  • 35% token supply freely tradeable with no lock
  • Smart contracts: 5 on-chain issues remain unfixed
  • New features deployed without security review

Risk Assessment: HIGH (regressed from MEDIUM-HIGH post-patch)

Security Score: 4.5/10 (down from 5.5/10 after patches, up from 2/10 at initial audit)

The platform should not go to production until:

  1. Stop storing user wallet private keys (NEW-03) — use API Wallet delegation only
  2. Restore Cloudflare protection on /api/miniapp/* routes (NEW-01)
  3. Lock liquidity via Team.Finance or equivalent — minimum 12 months
  4. Transfer contract ownership to a multisig with timelock (C12, H15)
  5. Redeploy smart contracts with SafeERC20 and fee fixes (C09, H07-H09)
  6. Verify parentAddress ownership with EIP-712 signatures (NEW-05)
  7. Remove production body logging (NEW-04)
