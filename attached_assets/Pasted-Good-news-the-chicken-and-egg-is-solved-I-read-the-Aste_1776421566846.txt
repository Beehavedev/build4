Good news — the chicken-and-egg is solved. I read the Aster docs.
There is NO vault contract to interact with. Aster deposits work 
like a CEX: each user gets a personal deposit address. You send 
USDT to it, Aster detects it, account is created automatically.

THE CORRECT FULL IN-APP FLOW:

═══════════════════════════════════════════════════
STEP 1: Get the user's Aster deposit address
═══════════════════════════════════════════════════

Call this PUBLIC endpoint (no auth needed):
GET https://www.asterdex.com/bapi/futures/v1/public/future/aster/deposit/assets?chainIds=56&networks=EVM&accountType=perp

Then to get the SPECIFIC user deposit address, call:
GET /fapi/v3/capital/deposit/address 
params: { coin: "USDT", network: "EVM" }
signed with: user + signer + ASTER_AGENT_PRIVATE_KEY

PROBLEM: This signed call returns 403 for new users because 
they don't have an Aster account yet.

REAL SOLUTION: The deposit address for a new user IS their 
own wallet address. Aster on BSC works by the user sending 
USDT to their own registered address. When a brand new user 
sends USDT to their BUILD4 wallet address on BSC, and that 
address has never interacted with Aster before, we need to 
bootstrap the account.

The bootstrap is done by calling approveAgent first — but 
approveAgent also fails for brand new wallets.

ACTUAL CORRECT SOLUTION based on docs:

The deposit address Aster gives you IS a separate address 
they assign. But for NEW users who have never used Aster, 
the account creation happens automatically when they first 
SEND a transfer to Aster via the on-chain transfer method.

Looking at the deposit docs:
curl 'https://www.asterdex.com/bapi/futures/v1/public/future/aster/deposit/assets?chainIds=56&networks=EVM&accountType=perp'

This returns the USDT contract address for deposits.
The actual DEPOSIT is: user sends USDT to Aster's 
designated deposit address for their account.
